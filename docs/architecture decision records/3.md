# ADR-003 — Light CQRS read model for order status

- **Status:** Accepted
- **Date:** 7.13.2026
- **Deciders:** Robert Felkins
- **Related:** ADR-002 (event log), ADR-001 (saga)

## Context

The customer status view polls for one order's current state every couple of seconds, and the ops view lists all non-terminal orders. Serving those reads from the append-only event log (ADR-002) would mean replaying an order's full event stream on every poll, and there is no efficient "list all active orders" query against a store partitioned by `OrderId`. Reads and writes have genuinely different shapes here: writes are an immutable event stream, reads want the latest projected state, fast.

## Decision

Apply a **light CQRS** split. Writes go through the domain and the event log; a **projection of current order status** is maintained in **Redis** and serves all status reads. The saga updates the read model on every transition. The read model exposes `GetStatusAsync(orderId)` and `ListActiveAsync()` (orders not yet `Confirmed` or `Failed`).

This is deliberately **not full event sourcing**. The read model is a convenience projection that can be rebuilt from the event stream — it is a cache of current state, not the system of record for every aggregate.

## Consequences

**Positive**

- Status polling is a single fast key lookup, not a stream replay. The ops "active orders" list is a cheap set read.
- The read model is disposable: it can be dropped and rebuilt from the Cosmos event log, so it never becomes a second source of truth to reconcile.
- The write side (event log) and read side (Redis) can evolve independently.

**Negative (accepted)**

- **Eventual consistency between write and read sides.** There is a brief lag between an event being appended and the read model reflecting it. A poll can momentarily show a slightly stale state. For an async workflow the customer is already polling, this is acceptable.
- **A projection to maintain.** Every new state or field that the UI needs must be written into the read model as well as the event log — two places to update, and a rebuild routine to keep correct.
- **Not "real" CQRS/ES.** Purists may note the read model isn't derived by a formal projection host replaying events; here the saga writes it inline. We accept the simpler approach because the POC doesn't need independent projection scaling.

## Alternatives considered

- **Full event sourcing of every aggregate**, with all reads served by replaying/projecting events. Rejected as over-engineering for the POC. It would impose an event-sourced model on inventory and payment (which are better served by ordinary relational rows) and add a projection-host mechanism the demo doesn't need. Right-sizing means using the read model only where it earns its keep — order status — not everywhere.
- **No read model; read status directly from the event log.** Rejected. It makes per-poll reads expensive and provides no good "list active orders" path for the ops view.
- **Serve status from the relational store.** Rejected — order state lives in the event log, not SQL; adding an order-status table in SQL would create a third place order state is written.
