# ADR-007 — A recovery sweeper instead of an outbox

- **Status:** Accepted
- **Date:** 7.14.2026
- **Deciders:** Robert Felkins
- **Related:** ADR-001 (saga, compensation only), ADR-002 (event log)

## Context

Everything in OrderFlow that is driven by a *message* recovers on its own. A handler throws, the broker redelivers, and after the delivery count is exhausted the message dead-letters where a human can see it. That is the whole reason the architecture tolerates at-least-once delivery.

`OrderBusinessManager.PlaceAsync` is not driven by a message. It is driven by HTTP, and it gets exactly one attempt. It does three things in a row:

1. append `OrderPlaced` to the event log,
2. project the order into Redis,
3. send `ReserveInventory`.

If step 3 throws, the order **exists**, is `Placed`, is sitting in the active list — and **nothing is listening to it, and nothing ever will be.** No message was lost, so no dead-letter queue shows anything. The customer sees a 500, re-posts, and gets a *second* order. It is the only failure in the system that is both silent and permanent, and it is at the very front door.

The textbook fix is a **transactional outbox**: write the command into the same transaction as the event append, and let a separate dispatcher drain it. That is correct, and it is a meaningful amount of machinery — an outbox table, a dispatcher process, a claim/lease protocol, and a delivery-ordering story — for a POC whose subject is compensation, not delivery guarantees.

## Decision

Add an **order recovery sweeper**: a `BackgroundService` in the Order API that periodically finds non-terminal orders which have stopped moving, and **re-sends the command their current state is waiting on**.

- `Placed` → re-send `ReserveInventory`
- `Reserved` → re-send `ChargePayment`
- `Paid` → re-send `DispatchFulfillment`
- anything else → nothing to do

What makes this safe is a property the system already had: **every command carries a deterministic `MessageId`** derived from `(OrderId, message type)`. So a command that *did* get through is deduped by the receiver's `(ConsumerName, MessageId)` guard, and re-sending it costs nothing. A command that never got sent is simply sent. The sweeper does not need to know which case it is in — and that is the entire trick.

The last hold-out was `ReserveInventory` itself, which was being sent with a random id. It now uses the same deterministic id the sweeper will use. Without that, every sweep would have re-reserved stock.

## Consequences

**Positive**

- The one permanently-stranding failure in the system becomes a **delay** rather than a lost order.
- It costs one class. No outbox table, no dispatcher, no lease protocol.
- It generalises: it recovers a lost `ChargePayment` or `DispatchFulfillment` too, not just the front-door case it was built for.
- It gives the ops view something true to show — `GET /api/Orders/stuck` is the same query the sweeper runs.

**Negative (accepted)**

- **It is not an outbox and does not pretend to be.** There is still a window in which the event is appended and the command is not sent; the sweeper closes it *eventually* (default: within a minute), not atomically. A system where that minute matters needs the outbox.
- **It re-sends on a timer, so recovery is bounded by the sweep interval**, not by the failure. An order stuck at 12:00:01 is recovered at 12:00:30, not immediately.
- **It cannot fix an order whose reply event dead-lettered**, and deliberately does not try. That failure is *visible* — it is in the dead-letter queue — and it wants a human, not a retry loop. A sweeper that quietly re-drove dead-lettered work would destroy the evidence the ops view exists to show. Recovery and diagnosis are different jobs.
- **It duplicates one piece of the saga's knowledge**: which command each state is waiting on. That mapping now exists in two files, and if the saga's steps change, the sweeper must change with them. This is a real maintenance hazard and is called out in the code.

## Alternatives considered

- **A transactional outbox.** The right answer for production, and the one to reach for the moment delivery guarantees are the subject rather than compensation. Rejected here on cost, not correctness.
- **Retry the send inside `PlaceAsync`.** Rejected: it only narrows the window. A retry loop that exhausts still leaves the order stranded, and a process crash mid-loop leaves it stranded with no loop.
- **Fail the HTTP request and delete the order if the send fails.** Rejected — that is a compensating write to an *append-only* event log, which ADR-002 exists to forbid. It also loses the record that the customer tried.
- **Do nothing and let the ops view show the stuck order.** Rejected: it makes an operator do by hand, every time, the one thing a computer can do reliably.
