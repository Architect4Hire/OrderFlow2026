# ADR-002 — Cosmos DB for the append-only order event log

- **Status:** Accepted
- **Date:** 7.13.2026
- **Deciders:** Robert Felkins
- **Related:** ADR-001 (saga), ADR-003 (read model)

## Context

Every meaningful transition in an order's life — placed, reserved, paid, dispatched, confirmed, failed, plus the compensations — needs to be recorded durably so the saga can recover its state after a restart and so we have an audit trail of what actually happened to an order. This log is written once per event and never modified. It is read back by the key of a single order (the correlation id), effectively always "give me the full history of order X."

We need a store whose shape matches that access pattern: high-volume appends, reads scoped to one partition key, no updates.

## Decision

Use **Cosmos DB** as the append-only order event log, **partitioned by `OrderId`**. Events are written through an `IOrderEventStore` abstraction with two operations — `AppendAsync` and `ReadStreamAsync` — and are never updated or deleted. Because each order is its own partition, reading an order's full stream is a single-partition read rather than a cross-partition scan.

For the POC this runs against the **Cosmos DB emulator** (ADR-005), against the real Cosmos SDK, so moving to a live account is a configuration change.

## Consequences

**Positive**

- The store's strengths line up with the workload: append-heavy writes, partition-keyed reads, schemaless event payloads that can evolve without migrations.
- Partitioning by `OrderId` keeps each order's stream on one physical partition, so the hot-path read (rebuild/inspect one order) is fast and cheap.
- An immutable log is a natural audit trail and the foundation for rebuilding the read model (ADR-003).

**Negative (accepted)**

- **Polyglot persistence cost.** We now run two database technologies (Cosmos here, SQL in ADR for inventory/payment). That is more operational surface, two sets of skills, and two connection stories. We accept this because forcing both workloads into one store would mean fighting it in one direction or the other.
- **Cross-order queries are deliberately awkward.** Anything that needs to scan across many orders does not belong against this store — it goes through the read model or a separate projection. Querying the event log across partitions on the hot path is explicitly disallowed.
- Cosmos request-unit provisioning and cost modeling is a real concern in a live deployment, deferred here by using the emulator.

## Alternatives considered

- **A single relational store for everything** (events + inventory + payments). Rejected. An append-only event stream keyed by order is not what a normalized relational schema is best at; modeling an evolving event envelope in rigid tables invites schema churn, and the write pattern is a poor fit. Using one store would optimize for neither workload.
- **Event sourcing on a purpose-built event store (e.g. EventStoreDB).** Rejected as over-scoped for the POC — it adds another infrastructure dependency and a heavier programming model than the light-CQRS approach in ADR-003 requires. Cosmos gives us the append-only stream we need without committing to full event sourcing everywhere.
