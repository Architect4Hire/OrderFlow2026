# ADR-001 — Saga with compensation for the order workflow

- **Status:** Accepted
- **Date:** 7.13.2026
- **Deciders:** Robert Felkins
- **Related:** ADR-002 (event log), ADR-003 (read model)

## Context

An OrderFlow order touches four independently deployed services in sequence — Inventory reserves stock, Payment charges, Fulfillment dispatches, Notification informs the customer — each owning its own data store. There is no shared database and no single transaction that can span all of them.

The workflow can fail partway through at every step: payment can decline after stock is reserved, fulfillment can fail permanently after payment succeeds, a message can be delivered twice. Whatever we choose has to guarantee that a partial failure never leaves the system in an inconsistent state — most importantly, that stock is never left reserved for an order that will never ship, and money is never captured for an order that cannot be fulfilled.

## Decision

Coordinate the workflow as a **saga**: a sequence of local transactions, each with a defined **compensating action** that semantically undoes it. The Order service is the sole orchestrator. It advances the order through its states in response to reply events, and on any failure it issues the compensations needed to unwind the work already done:

- Payment declines after reservation → `ReleaseInventory`, mark `Failed`.
- Fulfillment fails permanently after payment → `RefundPayment` + `ReleaseInventory`, mark `Failed`.
- Inventory rejects up front → mark `Failed` immediately (nothing to compensate).

Every transition is appended to the order event log and reflected in the read model, so the current saga state survives a restart. Compensations are idempotent: a redelivered failure event does not re-run them.

## Consequences

**Positive**

- No distributed transaction, no cross-service lock, no blocking two-phase coordinator. Services stay independent and available.
- Failure handling is explicit and auditable — each compensation is a named action in one place (the saga), not an implicit rollback.
- The model degrades gracefully: a slow or briefly unavailable service delays an order, it does not corrupt it.

**Negative (accepted)**

- The system is **eventually consistent**, not atomic. There is a window where inventory is reserved but payment has not yet resolved. Downstream reads must tolerate in-flight orders.
- Compensation is **semantic, not perfect**. A refund is a new transaction, not a true rollback; a released reservation was genuinely held for a period. We accept that "undo" means "compensating action," not "as if it never happened."
- The developer carries the burden of designing a correct compensation for every step and making it idempotent. This is the hardest part of the build — and the part most worth reviewing.

## Alternatives considered

- **Two-phase commit / distributed transaction.** Rejected. It would require a transaction coordinator spanning heterogeneous stores (Cosmos, SQL, a message broker), introduce cross-service locking and head-of-line blocking, and couple the services' availability together — the opposite of what an event-driven design is for. The operational and latency cost is not justified for an order workflow that tolerates eventual consistency.
- **A single service owning one database for the whole workflow.** Rejected. It would make the "distributed" nature of the demo disappear and wouldn't represent the real-world modernization scenario this reference architecture exists to show. (See ADR-004 for where we _did_ choose not to decompose.)

## Verification

**Verified:** `Orders.API/Managers/Saga/OrderSaga.cs` — `OnPaymentDeclinedAsync` (release), `OnFulfillmentFailedAsync` (refund **and** release), `OnInventoryRejectedAsync` (fail, no compensation). The terminal guard is `LoadAsync`, which returns `null` for a `Confirmed` or `Failed` order so a redelivered event is a no-op. No 2PC, `TransactionScope`, or distributed coordinator appears anywhere in the solution. The compensation ordering — every outbound message sent **before** the terminal state is recorded — is asserted in `tests/OrderFlow.UnitTests/OrderSagaTests.cs`, which also proves a redelivered `PaymentDeclined` releases stock exactly once.

**Open mismatch — "the Order service is the sole orchestrator" is true; "the saga is" is not.** At service granularity the claim holds: no other service issues a workflow command. But *within* the Order service, workflow knowledge now lives in **three** files, not one. `OrderBusinessManager.PlaceAsync` decides that a placed order's first step is `ReserveInventory`, and `OrderRecoveryManager.CommandFor` re-encodes the whole state→command mapping in order to re-drive stuck orders (ADR-007). The saga is the sole *decider*, but it is no longer the sole *knower*. This is the exact leak diagnostic I6 is written to catch, and reconciling it — by moving the start step onto the saga as a `StartAsync` — is a separate decision, deliberately not taken here.
