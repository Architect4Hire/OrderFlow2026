# ADR-005 — Aspire-local emulation with real Azure SDKs

- **Status:** Accepted
- **Date:** 7.13.2026
- **Deciders:** Robert Felkins
- **Related:** ADR-002 (Cosmos), design principle "local by default, real by SDK"

## Context

This is a portfolio/reference build that must be runnable by anyone, on a laptop, with no cloud account and no spend — while still being a credible demonstration of Azure-native architecture. The tension is between "cheap and runnable" and "real enough to be believable." A demo that quietly swaps out real messaging or storage semantics for in-memory fakes invites the fair criticism that it wouldn't survive contact with actual Azure.

## Decision

Run the entire system on **Aspire-orchestrated local emulators** — SQL Server in a container, the Cosmos DB emulator, Redis, and the Service Bus emulator — but write all code against the **real Azure SDKs** (`Azure.Messaging.ServiceBus`, the Cosmos SDK, etc.). Aspire's AppHost wires the resources; the services reference them by connection name. Going live is a **configuration change** (point the SDK at a live account / namespace), not a rewrite. Emulator-vs-live is marked at each resource with a `// TODO: swap emulator for live account via config` note.

## Consequences

**Positive**

- **Zero cloud spend** to build, run, or demo. The whole thing comes up with `dotnet run` on the AppHost.
- The code is **honest**: it uses real messaging and storage semantics (queues, topics, partitioned Cosmos, Redis), so the demo isn't a simulation dressed up as Azure. The swap-to-live claim is credible because there's nothing to rewrite.
- Aspire gives a single front door — the dashboard — for logs, traces, and resource health, which doubles as the observability demo surface.

**Negative (accepted)**

- **Emulators are not identical to the cloud.** The Cosmos and Service Bus emulators have known gaps and behavioral differences from live Azure (feature coverage, throughput/throttling behavior, some API surface). A green local run does **not** prove production-readiness; a real deployment still needs its own validation.
- **Local resource weight.** Running SQL, Cosmos emulator, Redis, and Service Bus emulator together is demanding on a laptop — startup time and memory are non-trivial.
- The `// TODO` swap points are real work in a live engagement (networking, identity, provisioning, cost/throughput tuning) that the POC intentionally defers.

## Alternatives considered

- **Provision live Azure resources for the demo.** Rejected. It imposes cost and an Azure account on anyone wanting to run the project, adds provisioning/identity/networking setup, and undermines the "runnable by anyone, zero spend" goal — for no demonstrative gain, since the SDK code is identical either way.
- **In-memory / mocked infrastructure** (fake bus, in-memory stores). Rejected. It would be cheaper and lighter but dishonest: it changes the semantics the system is supposed to demonstrate (at-least-once delivery, partitioned reads, dead-lettering) and invites exactly the "this wouldn't work for real" critique this build is meant to preempt.

## Verification

**Verified by inspection:** `AppHost/Program.cs` — SQL Server as a container with a persistent volume, `AddAzureCosmosDB(...).RunAsEmulator(...)`, `AddRedis(...)`, `AddAzureServiceBus(...).RunAsEmulator()`. The services consume them through the real SDKs (`Azure.Messaging.ServiceBus`, the Cosmos SDK, `StackExchange.Redis`, EF Core + `Microsoft.Data.SqlClient`), resolved by Aspire connection name — there is no hand-rolled fake bus or in-memory store anywhere in `src/`. The `// TODO: swap RunAsEmulator for live account via config` markers are present on both Azure resources. The claim that going live is a configuration change rather than a rewrite is credible: no service names a broker, a database, or a connection string.

**The zero-fakes rule held even under test pressure**, which is the strongest evidence for it. The unit suite deliberately does **not** use EF InMemory: it enforces neither row versions nor unique indexes, so a green test against it would have *implied* the oversell guard and the double-charge guard worked while proving nothing of the sort. Those two guarantees are made by the database, so they are proven in `tests/OrderFlow.IntegrationTests`, against the real containers — which is precisely the "semantics you cannot fake" argument this ADR makes, arriving as a practical constraint rather than a principle.

**Open — "the whole thing comes up with `dotnet run` on the AppHost" is not yet demonstrated, and was untrue for most of the build.** `AddJavaScriptApp("web", "../OrderFlow.Web", "start")` resolves its directory eagerly, and `src/OrderFlow.Web` does not exist until Part G — so from the moment the AppHost was written until that was found, the host would have failed at startup every single time, while `dotnet build` stayed green throughout. The web resource is now commented out until the Angular workspace lands, and the five services and four backing resources are wired and compiling. **But this has still not been executed end-to-end**, because the machine this was verified on has no running Docker daemon. Until someone starts it and watches five services reach Healthy, the headline claim of this ADR is inspected, not proven. A green build is not a running system.
